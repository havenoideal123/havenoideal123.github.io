这道题需要使用glibc2.23，如果没有搞对版本后续利用会出问题，建议先用ldd确认好版本再开始

WP

1.将程序丢入ida，程序逻辑较为简单，和前面做过的gundam之类的类似，可以推测出结构体：1.序号2大小3地址

可以看到在 fill 函数中可以输入 chunk 的 index 并填充自定义长度的内容，存在堆溢出漏洞。而dump函数是直接输出size的大小的内容，

因此思路很简单：堆溢出泄漏信息，dump输出，然后利用fastbin的攻击手法getshell

我们将关键点进行列举：

  • 存在堆溢出
  • 存在读内容（读再衍生一下就是泄露）
  • 保护措施全开

不难发现，本题的关键就是能否泄露出信息

想一下堆利用中能泄露信息的，第一反应就是Unsorted Bin Leak

而Unsorted Bin Leak能实现的关键就在：

  • 是否存在两个及以上的指针指向同一个chunk
  • 是否存在chunk的fd、bk指针信息读取的方式

很明显:

  • 虽然不能Double Free（因为 free函数之前存在检查（前8字节部分的值是否为1）），但是由于本题存在堆溢出，基本可以玩转Fast Bin里面的攻击手段，因此，可以通过伪造Fast Bin序列（将未被free的目标chunk放入bin序列，接着再申请chunk，即完成双指）来实现chunk的双指
    -Dump函数完美满足第二个条件

既然能实现Unsorted Bin Leak，即能完成libc基址的泄露

我们上面提到过“玩转Fast Bin里面的攻击手段”，很容易想到House Of Spirit，即任意写

首先，我们先确定chunk的数目：

  • House Of Spirit等Fast Bin利用都需要使Fast Bin中至少有两个chunk(chunk1,chunk2)
  • 本题的双指需要在Fast Bin中有两个chunk的前提下，再要一个未被free的chunk(chunk4)
  • 为了使得堆溢出更加灵活实用，我们在第一条提到的两个chunk前再加一个chunk(chunk0)，再第二条提到的一个chunk前再加一个chunk(chunk3)

注意：这个题目不涉及tacache技术，这里是glib2.23

关于大小的选择：

  • chunk0、3仅用于堆溢出，大小其实无所谓，只是尽量对齐
  • chunk1、2需要放入Fast Bin，不能过大，也尽量满足对齐
  • chunk4，最终需要放入unsorted bin，需要满足“释放（free）的chunk大小不属于fast bin，并且该chunk不和top chunk紧邻”，先满足前半句（64位Fast Bin中能存放的默认最大值为(64 * SIZE_SZ / 4) = (64 * 8 / 4) = 128B = 0x80，但是我们是申请0x80，也就意味着size的大小为0x90>0x80即不满足Fast Bin），后半句后续再处理

根据思路，为了实现双指，我们需要让两个chunk先进入Fast Bin

free(p,1)
free(p,2)

此时内存：heap->chunk2->chunk1

现在fastbin中已经有chunk2和chunk1了。我们有4个结构体。现在我们需要将chunk2改为指向chunk4，可以利用堆溢出。我们设计的是chunk0用于溢出，那么就从chunk0开始看。所以payload1就是为了覆盖chunk0-》chunk4的内容，将chunk2的fd覆盖成了chunk4的地址。这样fastbin中heap->chunk2->chunk4

有两个问题：

  • Fast Bin符合LIFO，即后进先出，那么先出的就是chunk2而非chunk4，因此我们需要Allocate两次
  • malloc的检查

malloc的检查什么意思呢？

我们知道，在Fast Bins中，每个Fast Bin里面的chunk的大小都是一致的

换言之，原先chunk1的size大小是0x21（1是三位flag，不会计入实际大小，因此该chunk的大小为0x20），那么chunk1就会被存储在chunk大小都为0x20的Fast bin中

那当我们再次malloc一个0x10大小的chunk的时候（malloc指定的大小是不含chunk header的（占0x10）），就会去chunk大小都为0x20的Fast bin中去查找，但是此时找到一个大小为0x90的chunk，malloc就会检查出异常，因而程序终止

因此，我们在申请之前，再通过堆溢出（这次通过chunk3来，因为近），修改chunk4的size字段，将其修改为0x21，从而绕过判断

即，我们先将chunk4的大小改成0x21。然后再申请它，此时出现了双指向。因为chunk2对应index2，他的fd是chunk4，而chunk4本身有个index4也是指向chunk4的，这样当chunk4走到unsortedbins时，我们还是可以用index2来找到它

先让chunk4进入Unsorted Bin

之前也提到过，要让其进入Unsorted Bin需要满足两个条件：

  • 释放（free）的chunk大小不属于fast bin的范围
  • 该chunk不和top chunk紧邻时

关于第一条，我们只需要将其大小改回去（0x21 -> 0x91），然后free即可

关于第二条，由于chunk4是最晚申请的，他就挨着top chunk，为了满足条件二，我们可以再申请一个chunk5，让chunk4与top chunk分开

chunk4成功进入unsortedbin，且出现特征：“如果usorted bin中只有一个bin ，那么它的fd和bk指针会指向同一个地址，就是unsorted bin链表的头部”

此时我们已经能够通过index2 拿到泄漏的地址，用这个地址能够获得libc基地址

__malloc_hook

目标：修改__malloc_hook为one_gadget的地址
手段：House Of Spirit

因此，我们需要再次在Fast Bin中放入chunk，然后修改fd指针完成链接的重定向

关键点来了，我们上面讲过：

  • 同一链接下的chunk的大小相同
  • malloc会检查bin中的数据大小

我们最终的目的是在__malloc_hook附近创建chunk然后写入数据，实现任意地址写入

而实现任意地址写入的方法就是修改链条的指向（指向__malloc_hook附近的那块地址，作为伪chunk）

因此，我们需要通过__malloc_hook附近的伪size字段的大小来决定我们需要申请的chunk的大小是多少

1. 动机：为什么要往 __malloc_hook 写东西？

  • 最终目标 ：拿 Shell。
  • 方法 ：如果我们将 __malloc_hook 的内容修改为 one_gadget 的地址，那么当程序下一次调用 malloc() 时，它就不会去分配内存，而是直接跳转去执行 one_gadget，从而给我们 Shell。
  • 实施路径 ：我们想让 malloc() 返回一个指针，这个指针刚好指向 __malloc_hook 的位置。这样我们就可以像往普通堆块写数据一样，往 __malloc_hook 里写地址了。

2. 阻碍：Malloc 的安全检查机制

你不能随便告诉 Fastbin：“嘿，下一个空闲块在 __malloc_hook 的地址，你去拿吧。”

当你修改了 Fastbin 中某个 Chunk 的 fd 指针指向 __malloc_hook 附近后，当你试图通过 malloc() 把这个伪造块申请出来时，Glibc 会做一个 致命的检查 ：

Fastbin Size Check :
系统会去你指定的那个地址（即伪造块的头部），读取那里的 size 字段。
系统会检查：“这个 size 的大小，是否真的属于当前这个 Fastbin 链表？”

举个例子（灾难现场）：

  1. 你申请的是 0x60 大小的 Chunk（对应 0x70 的 Fastbin）。
  2. 你把 fd 指向了 __malloc_hook。
  3. 当你 malloc(0x60) 时，系统跑去 __malloc_hook 那个地址看了一眼。
  4. 系统发现那里全是 0（因为 __malloc_hook 初始是空的，周围可能也是空的）。
  5. 系统怒了：“你要申请 0x70 的块，指过来的这个地方 size 却是 0？ 报错！malloc(): memory corruption (fast) ”

3. 手段：寻找“天选之子”伪造 Size

既然系统要检查 Size，而我们又不能直接去 libc 的代码段里写一个 0x71 放在那里（因为那里通常不可写，或者还没拿到写权限）。

怎么办？我们只能“借用”现有的数据。

我们需要在 __malloc_hook 的前面一点点的位置，寻找有没有现成的字节，长得像一个合法的 size。

经典的 0x7f 字节

在绝大多数 64 位 Glibc 版本中，__malloc_hook 前面是 __memalign_hook 和 __realloc_hook。在这些函数指针附近，通常会有一些库的地址。

而在 __malloc_hook - 0x23 这个神奇的偏移处，通常由于地址的字节排列，会天然存在一个字节： 0x7f 。

让我们看看内存大概长什么样（以 GDB 视角）：

Plaintext

如果你把 fake chunk 的头部地址定在 0x7ffff7dd1b0d (这行数据的开头)：

  1. Size 域位置 ：chunk_ptr + 0x8。但这里我们通常利用 错位 。
  2. 更常用的偏移 ：通常我们找 __malloc_hook - 0x23。

核心逻辑来了：
如果在 __malloc_hook 附近的某个位置，这 8 个字节读起来像是 0x000000000000007f。
那么：

  • 这个 0x7f 可以被当做 chunk 的 size。
  • 0x7f 对应的十进制是 127。
  • 在 Fastbin 中，后三位标志位被忽略，所以它被视为大小为 0x70 的 Chunk。

4. 为什么这决定了你申请 Chunk 的大小？

这解释了你文中那句晦涩的话：

“因此，我们需要通过__malloc_hook附近的伪size字段的大小来决定我们需要申请的chunk的大小是多少”

逻辑闭环：

  1. 我们在 __malloc_hook 附近发现了一个天然存在的 0x7f。
  2. 这意味着我们只能伪造一个 0x70 大小 的 Chunk。
  3. 因为 Fastbin 是严格分类的， 0x70 的伪造块，必须放在 0x70 的 Fastbin 链表里 。
  4. 为了操作 0x70 的 Fastbin 链表，我们的代码（exploit script）里 必须申请 malloc(0x60) （因为 0x60 用户数据 + 0x10 头 = 0x70）。

如果你申请 malloc(0x50)，系统会去查 0x60 的链表，然后发现你伪造的那个地址 size 是 0x7f（属于 0x70 组），直接报错。

于是乎我们找到了这个&__malloc_hook -23的位置，然后对应size为0x7f，因此我们需要一个0x70的chunk，计算这个位置和libc基地值的相对偏移，然后得到绝对地址。这里为什么是申请的0x60?，因为头部还有0x10

此时，chunk2 fd依然是指向chunk4，我们申请然后释放的也是chunk4，因为它在unsorted bin中，这里将它进行了切分，但是序号没变（*index4还是指向chunk4头，只是大小变成了0x71，剩下了一小块留在unsorted）。修改index2，也就修改了chunk4的头部，由于chunk4被释放后在fastbin中，所以它的fd被修改成了&__malloc_hook-23的位置，我们两次申请后，拿到了这个位置。我们对中间的内容进行填充，直到____malloc_hook的起始位置，这里需要填入one___gadget找到的命令执行相对地址，写入后，再次申请任意大小内存，只要触发了malloc函数，就会触发malloc_hook函数，从而getshell
